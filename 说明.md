# `FilterWrapperHelper` 使用说明文档

## 1. 简介
`FilterWrapperHelper` 是一个基于**自定义注解**的 MyBatis-Plus 查询条件（Wrapper）自动组装工具类。
它的主要作用是**消除我们在 Service 层手写大量 `if-else` 来拼接 `QueryWrapper` 的重复代码**。只需在 DTO/VO 类的字段上加上 `@FilterFiled` 注解，工具类就能根据注解自动解析并生成对应的 MyBatis-Plus 查询条件，不仅支持常规的精确、模糊匹配，还原生支持 MySQL 的 JSON 字段检索。

**核心优势**：
- 🚀 **极简代码**：抛弃冗长的 `wrapper.eq().like()` 代码，一行调用搞定所有筛选。
- 🛡️ **安全防注入**：底层使用 MyBatis 预编译参数 `{0}`，有效杜绝 SQL 注入。
- 🧬 **支持继承**：完美支持继承自父类（如 `BaseEntity`）的筛选字段。
- 📦 **复杂查询支持**：内置支持 JSON 数组、JSON 对象的复杂查询。

---

## 2. 核心功能及注解说明

### `@FilterFiled` 注解参数解析：
| 属性 | 说明 | 默认行为/示例 |
| :--- | :--- | :--- |
| `type` | 筛选的匹配类型（对应 `FilterTypeEnum`） | `EQ` (等于), `LIKE` (包含), `LIST` (IN查询) 等 |
| `ifCamel2UnderLine` | 是否将驼峰字段名转为数据库下划线字段名 | `true` (如 `userName` -> `user_name`) |
| `listField` | 当 `type=LIST` 时，指定数据库对应的列名 | 如 `"status"` (用于 `status IN (...)`) |
| `jsonObjName` | JSON 对象在数据库中的实际字段名 | 针对 `JSON_OBJ` 类型的查询 |
| `aliasName` | JSON 内部对应的 key 名称 | 提取 JSON 内特定的值进行匹配 |

### `FilterTypeEnum` 支持的匹配类型：
- `EQ`：精确匹配（`=`）
- `LIKE`：全模糊匹配（`LIKE %xxx%`）
- `LEFT`：左模糊匹配（`LIKE %xxx`）
- `RIGHT`：右模糊匹配（`LIKE xxx%`）
- `LIST`：集合包含查询（`IN (v1, v2)`）
- `JSON_LIST`：JSON 数组交集查询（`JSON_OVERLAPS`）
- `JSON_OBJ`：JSON 对象属性查询（提取 JSON 指定 key 的值进行比较）
- `JSON_OBJ_LIST`：JSON 数组包含指定对象查询（`JSON_CONTAINS`）

## 3. 基础使用步骤

### 步骤 1：在查询对象 (DTO) 上添加注解
假设我们有一个用户查询请求实体 `UserQueryDTO`：

```java
@Data
public class UserQueryDTO {

    // 1. 姓名：全模糊匹配 (LIKE %name%)，字段自动转下划线 user_name
    @FilterFiled(type = FilterTypeEnum.LIKE, ifCamel2UnderLine = true)
    private String userName;

    // 2. 手机号：精确匹配 (=)
    @FilterFiled(type = FilterTypeEnum.EQ)
    private String phone;

    // 3. 状态集合：IN 查询 ( status IN (0, 1) )，指定数据库列名为 status
    @FilterFiled(type = FilterTypeEnum.LIST, listField = "status")
    private List<Integer> statusList;
    
    // 4. JSON 检索：假如数据库字段 tags 是 JSON 数组 ["vip", "new"]，查找包含特定标签的用户
    @FilterFiled(type = FilterTypeEnum.JSON_LIST, ifCamel2UnderLine = true)
    private List<String> tags;
}
```

### 步骤 2：在 Service/Controller 中一行调用
将查询对象传入 `FilterWrapperHelper`，直接获取 `QueryWrapper`。

```java
@Service
public class UserServiceImpl extends ServiceImpl<UserMapper, User> {

    @Autowired
    private FilterWrapperHelper<User> filterHelper;

    public List<User> getUserList(UserQueryDTO queryDTO) {
        // 核心：一句话自动生成 MyBatis-Plus Wrapper！
        QueryWrapper<User> wrapper = filterHelper.getWrapperbyFiledAnnotion(queryDTO);
        
        // 传递给 Mapper 执行查询
        return this.list(wrapper);
    }
    
    public List<User> getUserListLambda(UserQueryDTO queryDTO) {
        // 如果你喜欢用 LambdaQueryWrapper，也可以这样调用：
        LambdaQueryWrapper<User> lambdaWrapper = filterHelper.getLambdaWrapperbyFiledAnnotion(queryDTO);
        return this.list(lambdaWrapper);
    }
}
```

## 4. 进阶用法

### 4.1 结合时间段筛选 (`TimeFilterModel`)
如果你需要对特定的时间字段（例如 `update_time` 或 `create_time`）进行范围检索，可以传入 `TimeFilterModel`。

```java
public List<User> getListWithTime(UserQueryDTO query, TimeFilterModel timeModel) {
    // 默认按照 updateTime 字段进行时间范围 (ge, le) 查询
    QueryWrapper<User> wrapper = filterHelper.getWrapperbyFiledAnnotion(query, timeModel);
    
    // 如果你想指定其他时间字段（比如 create_time）：
    // QueryWrapper<User> wrapper = filterHelper.getWrapperbyFiledAnnotion(query, timeModel, "create_time");
    
    return this.list(wrapper);
}
```

### 4.2 基础审计字段筛选 (`BaseFilterModel`)
如果你的实体继承了 `BaseModel`（包含创建人、更新人、创建时间、更新时间），可以使用内置的 `getBaseFilter` 方法快速叠加这些通用条件的筛选：

```java
public List<User> getListByBaseInfo(BaseFilterModel baseFilter) {
    LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
    
    // 自动追加 create_user, update_user, create_time 范围等条件
    filterHelper.getBaseFilter(wrapper, baseFilter);
    
    return this.list(wrapper);
}
```

### 4.3 快速生成分页对象 (`Page`)
可以通过传入 `PageFilterModel` 快速获取 Mybatis-Plus 能够识别的分页 `Page` 对象：

```java
public IPage<User> pageUsers(PageFilterModel pageModel, UserQueryDTO queryDTO) {
    // 1. 获取分页对象 (默认防空处理，空则返回 1页 100条)
    Page<User> page = filterHelper.getPage(pageModel);
    
    // 2. 获取查询条件
    QueryWrapper<User> wrapper = filterHelper.getWrapperbyFiledAnnotion(queryDTO);
    
    // 3. 执行分页查询
    return this.page(page, wrapper);
}
```

## 5. 注意事项与最佳实践

1. **父类属性继承生效**：
   你可以放心地把 `@FilterFiled` 标在 `BaseDTO` 等父类中，工具类内置了 `ReflectionUtils.doWithFields`，能够成功解析继承下来的所有过滤字段。
2. **忽略空值**：
   如果传入的 DTO 字段值为 `null` 或 空字符串/空集合，工具类会自动忽略该字段，不会拼接到 SQL 中（避免出现 `WHERE name = null` 的死板情况）。
3. **MySql JSON 函数限制**：
   使用 `JSON_LIST` (底层依赖 `JSON_OVERLAPS`) 或 `JSON_OBJ_LIST` (底层依赖 `JSON_CONTAINS`) 时，请确保你的数据库版本为 **MySQL 8.0+**，否则部分较新的 JSON 检索函数可能在旧版数据库中不支持。
4. **防 SQL 注入保障**：
   工具类采用了 `{0}` MyBatis 原生预编译参数化注入（如 `CAST({0} AS JSON)`），处理特殊字符十分安全，无需担心 JSON 数据解析引发的 SQL 注入攻击。

# 1. `MongodbFilterHelper<T>` 使用说明文档

## 简介
`MongodbFilterHelper` 是针对 **Spring Data MongoDB** 封装的自动化查询构建工具。它通过解析 DTO 上的 `@FilterFiled` 注解，自动生成 MongoDB 的 `Criteria` 和 `Query` 对象。

## 核心特性
*   **正则安全**：自动对 `LIKE` 模糊查询内容进行 `Pattern.quote` 转义，防止正则注入攻击。
*   **分页纠正**：自动处理前端 1-indexed 页码与 MongoDB 底层 0-indexed 页码的转换。
*   **时间安全解析**：内置严谨的时间字符串转 `Date` 逻辑，杜绝解析异常导致的空指针。
*   **向后兼容**：保留了旧版拼写错误的接口代理，升级无感知。

## 快速上手

### 第一步：定义查询 DTO
```java
@Data
public class ArticleQueryDTO {
    // 精确匹配：分类 ID
    @FilterFiled(type = FilterTypeEnum.EQ)
    private String categoryId;

    // 模糊匹配：标题 (底层自动转为安全正则检索)
    @FilterFiled(type = FilterTypeEnum.LIKE)
    private String title;

    // 集合查询：标签 ID 列表 (IN 查询)
    @FilterFiled(type = FilterTypeEnum.LIST, listField = "tagIds")
    private List<String> tags;
}
```

### 第二步：Service 层调用
```java
@Service
public class ArticleService {
    @Autowired
    private MongodbFilterHelper<Article> mongoHelper;
    @Autowired
    private MongoTemplate mongoTemplate;

    public PageData<Article> search(ArticleQueryDTO dto, PageFilterModel page, TimeFilterModel time) {
        // 1. 自动组装查询条件 (Criteria)
        Query query = mongoHelper.getMongoQueryByAnnotation(dto, time, "createTime");

        // 2. 组装分页 (自动处理 1->0 页码转换)
        Pageable pageable = mongoHelper.getPageableByPageFilterModel(page);
        query.with(pageable);

        // 3. 执行查询
        List<Article> list = mongoTemplate.find(query, Article.class);
        long total = mongoTemplate.count(query, Article.class);

        // 4. 转化为统一分页结果对象
        PageImpl<Article> pageImpl = new PageImpl<>(list, pageable, total);
        return mongoHelper.getPageByPageImpl(pageImpl);
    }
}
```

# 2. `ESFilterHelper<T>` 使用说明文档

## 简介
`ESFilterHelper` 是专为 **Spring Data Elasticsearch** 设计的查询助手。它利用 ES 的 `CriteriaQuery` 体系，将注解逻辑转化为 ES 的布尔查询（Bool Query）逻辑，支持通配符和前缀检索。

## 核心特性
*   **语义转换**：将 `LIKE` 映射为 ES 的 `contains`，`LEFT` 映射为 `startsWith`，极大符合业务直觉。
*   **拆包助手**：针对 ES 特有的 `SearchHit<T>` 包装结构，提供一键拆解并转换为 `PageData` 的功能。
*   **Criteria 链式合并**：内部处理了 ES Criteria 的动态 `and` 合并逻辑，确保空条件时不会引发构建崩溃。

## 快速上手

### 第一步：定义查询 DTO
```java
@Data
public class LogESQueryDTO {
    // 关键字搜索：内容包含 (自动映射为 wildcard/contains 查询)
    @FilterFiled(type = FilterTypeEnum.LIKE)
    private String content;

    // 链路 ID 搜索：精确匹配
    @FilterFiled(type = FilterTypeEnum.EQ)
    private String traceId;

    // 级别筛选：IN 查询
    @FilterFiled(type = FilterTypeEnum.LIST, listField = "level")
    private List<String> levels;
}
```

### 第二步：Service 层调用
```java
@Service
public class LogService {
    @Autowired
    private ESFilterHelper<LogDoc> esHelper;
    @Autowired
    private ElasticsearchOperations esOperations;

    public PageData<LogDoc> searchLogs(LogESQueryDTO dto, PageFilterModel page) {
        // 1. 自动组装 ES CriteriaQuery
        CriteriaQuery query = esHelper.getESQueryByAnnotation(dto);

        // 2. 设置分页 (0-indexed 转换)
        query.setPageable(esHelper.getPageableByPageFilterModel(page));

        // 3. 执行 ES 检索
        SearchHits<LogDoc> searchHits = esOperations.search(query, LogDoc.class);

        // 4. 借助 SearchHitSupport 转换为 SearchPage
        SearchPage<LogDoc> searchPage = SearchHitSupport.searchPageFor(searchHits, query.getPageable());

        // 5. 一键转换为项目统一的 PageData 格式
        return esHelper.getPageBySearchPage(searchPage);
    }
}
```

## 3. 常见问题 (FAQ)

**Q: 为什么我设置了 `pageNum=1`，MongoDB/ES 却查不到数据？**
A: 这是因为 Spring Data 底层是从 `0` 开始计数的。请务必使用工具类提供的 `getPageableByPageFilterModel` 方法，它会自动完成 `pageNum - 1` 的逻辑转换。

**Q: 我的 DTO 继承了一个基础类，里面的字段能被解析吗？**
A: 可以。所有 FilterHelper 均使用了 `ReflectionUtils.doWithFields` 递归扫描父类 Field，完美支持继承体系。

**Q: LIKE 查询支持特殊字符吗？（针对 MongoDB）**
A: 支持。优化后的代码对用户输入进行了 `Pattern.quote` 处理，哪怕用户输入 `*+?` 等正则敏感字符，也会被当做纯文本进行模糊匹配，不会引发正则语法报错或安全风险。
